# PM 工作流方法論 Kit（Core）

> 這份是骨架，不含任何特定專案、特定公司、特定資料庫的細節。
> 拿到任何新專案，複製這個 `core/` 資料夾，
> 在 `instances/<專案名稱>/` 底下建一份新的 SETUP，就能開始用。

## 這份 Kit 解決什麼問題

用 AI 輔助開發時，常見的失敗模式是：**人提出模糊需求 → AI 直接動手做 → 做出來的東西範圍失控、驗收標準模糊、沒有人真的 review 過。**

這份 Kit 的核心主張是：**把「產出規格」「執行」「審查」拆成三個角色，即使只有一個人在做，也要用三種不同的思考模式分別走過一次。** 這不是流程官僚，是用結構換取品質跟可傳承性。

## 內容地圖

| 檔案 | 解決什麼問題 |
|---|---|
| `workflow-pipeline.md` | 需求怎麼變成可執行的規格、怎麼審查回傳結果 |
| `codex_spec_template.md` | 每次交派任務的規格書固定格式 |
| `handover_template.md` | 交接/中斷點紀錄的固定格式，讓任何人（含未來的自己）10分鐘內接手 |
| `data-cleaning-patterns.md` | 處理外部/半結構化資料時，普遍會踩到的陷阱模式 |
| `governance-checklist.md` | 「這件事該不該做」的判斷框架，跟具體案例分開 |
| `config-categories.md` | 一個新專案通常需要盤點哪幾類環境設定 |

## 使用方式

1. 新專案開始時，複製整個 `core/` 資料夾
2. 在 `instances/<專案名稱>/` 建立這個專案專屬的 `SETUP.md`、地雷記錄、判斷案例
3. `core/` 裡踩到「這個陷阱其實不是這個專案獨有，換個專案也會遇到」的坑，就把它從 instance 層搬進 core 層——**這個搬遷的判斷本身，就是這份 Kit 會越用越準的原因**

## 這份 Kit 想證明的事（給看到這份文件的人）

一個人能不能在專案結束、角色轉換、甚至職務調整，
留下一套「別人接得住」的方法論，這件事本身就是 PM 能力的具體展現——
不是靠職稱證明，是靠這份文件能不能真的讓下一個人少走彎路來證明。
