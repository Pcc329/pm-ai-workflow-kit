> `core/` 是骨架，不含任何特定專案、特定公司、特定資料庫的細節。
> 拿到任何新專案，先跑一次啟動訪談（見使用方式第 1 步），
> 在 `instances/<專案名稱>/` 底下產出一份 SETUP，就能開始用。

## 這份 Kit 解決什麼問題

用 AI 輔助開發時，常見的失敗模式是：**人提出模糊需求 → AI 直接動手做 → 做出來的東西範圍失控、驗收標準模糊、沒有人真的 review 過。**

這份 Kit 的核心主張是：**把「產出規格」「執行」「審查」拆成三個角色，即使只有一個人在做，也要用三種不同的思考模式分別走過一次。** 這不是流程官僚，是用結構換取品質跟可傳承性。

## 結構

```
core/        通用方法論（可攜帶，跨專案不變）
instances/   各專案的具體事實（SETUP、地雷紀錄、判斷案例）
             └ example-project/  一份用虛構專案填好的示範
```

## 內容地圖（core/）

| 檔案 | 解決什麼問題 |
|---|---|
| `project-kickoff-interview.md` | **入口**。貼給 AI 就能訪談新接手的人，產出專案 SETUP 初稿 |
| `workflow-pipeline.md` | 需求怎麼變成可執行的規格、怎麼審查回傳結果 |
| `ai-collaboration-roles.md` | 多 AI 工具偕同時，哪個工具站哪個崗、為什麼檢查要做兩道 |
| `codex_spec_template.md` | 每次交派任務的規格書固定格式 |
| `handover_template.md` | 交接/中斷點紀錄的固定格式，讓任何人（含未來的自己）10分鐘內接手 |
| `data-cleaning-patterns.md` | 處理外部/半結構化資料時，普遍會踩到的陷阱模式 |
| `governance-checklist.md` | 「這件事該不該做」的判斷框架，跟具體案例分開 |
| `config-categories.md` | 一個新專案通常需要盤點哪幾類環境設定 |

## 使用方式

1. **新專案開始時**：打開 `core/project-kickoff-interview.md`，把提示詞貼給任何 AI 工具，
   照著訪談回答——不用先讀完整個 Kit，訪談會帶你走完啟動流程
2. 把訪談產出的 SETUP 初稿存進 `instances/<專案名稱>/SETUP.md`，
   之後的地雷紀錄、判斷案例也都放在同一個資料夾
   （不確定填到什麼程度算夠？看 `instances/example-project/SETUP.md` 的示範）
3. 踩到「這個陷阱其實不是這個專案獨有，換個專案也會遇到」的坑，
   就把它從 instance 層搬進 core 層——**這個搬遷的判斷本身，就是這份 Kit 會越用越準的原因**

## 這份 Kit 想證明的事（給看到這份文件的人）

一個人能不能在專案結束、角色轉換、甚至職務調整，
留下一套「別人接得住」的方法論，這件事本身就是 PM 能力的具體展現——
不是靠職稱證明，是靠這份文件能不能真的讓下一個人少走彎路來證明。
