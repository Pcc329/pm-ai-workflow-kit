# 工作流程：需求 → 規格 → 執行 → 審查

## 整體流程

```
人描述需求
  → 規格撰寫者（可以是 AI 助手，也可以是你自己換一個角色思考）產出規格書
  → 執行者（可以是另一個 AI 工具、外包、或團隊成員）依規格執行
  → 執行者回傳「完成報告」（列出實際改了什麼、怎麼驗證的）
  → 審查者（原本提規格的人）對照規格逐項核對完成報告
  → 審查通過 → 正式提交（PR / 交付 / 上線）
  → 驗收 → 確認收尾
```

## 為什麼要拆成「產規格」「執行」「審查」三個角色

即使實際上只有你一個人（搭配一個 AI 工具）在做這件事，
**分開思考這三個角色，本身就是一種品質保證機制**：

- 執行者沒有直接決定「範圍」的權力——規格先寫清楚，執行時比較不會自作主張擴大或縮小範圍
- 規格把「防呆」（不該動的東西）寫清楚，能大幅降低執行者用「表面正確、實則跑偏」的方式達成任務
- 完成報告強制要求列出「實際改了什麼」，讓審查有東西可以核對，不是憑印象檢查
- 分階段留下紀錄，天然形成一條可回溯的軌跡，不需要事後才想起來要記錄什麼

## 規格書寫作規則（完整範本見 `codex_spec_template.md`）

1. **背景要寫清楚**：讓執行者理解「為什麼」，不是只給「要做什麼」——否則容易用錯的方式達成表面上對的結果
2. **每項改動用「現況 → 需求」對照寫**：具體到牽涉哪個檔案 / 哪個模組
3. **驗收標準要能勾選**：寫成可以打勾的條件句，不要寫模糊的形容詞（例如「效果要好」不是驗收標準，「回應時間 < 2 秒」才是）
4. **一定要寫「不要動的東西」**：這是防呆的關鍵段落，避免執行者在你沒注意到的地方順手做了不該做的更動
5. **驗證資料要固定用最新版本**：明確指定驗證時要用哪份資料 / 哪個版本，避免執行者拿舊的快取資料做出看似正確、實則跟現況脫節的驗證結果
6. **完成報告要求要固定寫**：改動清單、關鍵邏輯位置、驗證範例（含前後對比）、驗收標準逐項確認、版本控制資訊（branch/commit/PR 連結）——版本資訊這項容易被忽略，但沒有它，審查者無法真正對照到實際程式碼

## 版本控制與提交紀律

- 每個功能獨立開分支，不要把不相關的改動混在一起
- 不直接提交到主分支，除非是明確被授權的緊急修正
- 檔名/變數命名一律用英文，避免不同系統編碼不一致造成亂碼

## 執行前的保守原則

- **能不動的就不動**：介面、既有邏輯的變動範圍越小越好，避免不必要的關係人疑慮或範圍蔓延
- **底層資料優先於上層功能**：資料完整性、資料結構的正確性，優先於功能堆疊的速度
- **任何寫入型操作，先小規模試跑（DRY_RUN）**：用小樣本驗證、印出預覽結果、人工確認過一次，才切換成正式執行；牽涉刪除的操作，先備份再動手

## 對外溝通用詞的分寸

當分析結果要提供給其他人（同事、主管、外部對象）做決策依據時：

用詞要基於「系統實際記錄/驗證過什麼」，而不是引申成更強的關係或結論。
陳述「資料庫裡有這筆記錄」跟陳述「我們跟對方有實質往來」是兩種不同層級的斷言，不能混用。
這個原則同樣適用在對主管回報進度——陳述事實比修飾措辭更重要，尤其在事後可能被追問細節的情境下。
